# DSCraft — Unified Local-First ML Tooling Platform
## Consolidated Technical Architecture (v1 Scope)

*Internal planning document. Not customer-facing. Compiled from nine independent module research documents, five parallel deep-research passes resolving cross-module contradictions, and a structured scope-lock Q&A with the platform owner.*

---

## Part 1: Executive Summary & Scope

**DSCraft** is a unified, MIT-licensed, local-first ML tooling platform spanning the full workflow from data curation through model training, evaluation, and security testing — tabular AutoML, data cleaning, time-series forecasting, graph ML, computer vision, LLM fine-tuning, LLM/agent red-teaming, and agent/RAG benchmark evaluation. The reference environment is a high-memory Apple Silicon workstation (M4 Max class, 128GB unified memory), not a resource-constrained edge device or a cloud deployment target. The platform's own code is 100% MIT-licensed; any external model it bundles or recommends must carry business-safe licensing terms, enforced through a per-module allowlist policy.

This document exists because DSCraft did not start as one design. It started as nine independently written technical research documents, each proposing a standalone tool under its own internal "Lazy*" codename (LazyClean, LazyForecast, LazyGraph, LazyVision, LazyTune, LazyRed, LazyAgent, plus the AutoML and LazyEdge modules). Read side by side, those nine documents contradicted each other in ways that would have produced a broken or bloated product if built as originally specified: four incompatible strategies for isolating risky dependencies, no shared data or tensor representation across modules that manipulate largely the same underlying data, ONNX export logic independently duplicated across three-plus modules, and at least one fabricated tool citation in the security-testing research. Five parallel deep-research passes were run to resolve these contradictions on their technical merits, and the resulting options were then taken to the stakeholder in a structured Q&A to lock scope. This document is the synthesis of that process — the architectural decisions that survived, and the reasoning behind them.

**The v1 boundary, stated plainly**: v1 is local-only. There is no cloud deployment path and no remote inference endpoint as a first-class target anywhere in the platform. MPS (Apple Metal) is the primary backend; Linux+CUDA is a secondary target the platform is designed to accommodate but not primarily built or validated against in v1. The 128GB unified-memory reference machine means DSCraft can reasonably target ~70B-parameter-class quantized local LLMs — but only if those models are Mixture-of-Experts architectures with a small active-parameter footprint; a dense 70B model at that scale is impractically slow on this hardware, so default model guidance throughout the platform favors MoE over dense at large parameter counts. Memory-conscious fallback logic (OOM avoidance, static memory planning, downsampling) is retained across modules as a safety net for smaller machines, but it is not the primary design assumption — the primary assumption is that memory is abundant. There is no LLM router and no multi-provider abstraction anywhere in the platform; each module takes a bare-minimum "bring your own local model handle" approach with no shared interface yet, and this is a deliberate, explicit exclusion rather than a phase-1 gap.

**Explicitly out of v1**: edge/microcontroller compilation (LazyEdge) is deferred in full, because it is fundamentally a production/cloud-deployment concern that doesn't fit a local-dev-focused v1. Cloud and remote targets for the red-teaming and agent-evaluation modules are deferred — v1 covers local targets only. Formal typed interface contracts between modules (in the style of MLflow's model "flavors") are deferred as premature until at least two real modules actually need to exchange data. GPU-datacenter LLM serving and quantization tooling (vLLM, TensorRT-LLM, AutoAWQ-Marlin) is out of scope for the fine-tuning module in v1, replaced by local-serving formats (GGUF, MLX). These exclusions are not oversights; each reflects a scope decision made to keep v1 buildable, coherent, and genuinely local-first rather than a thin veneer over cloud-shaped assumptions. They are tracked as an internal roadmap in Part 6 of this document.

## Part 2: Shared Core Architecture (LazyCore)

### 2.0 Overview

LazyCore is the thin, shared substrate underneath all nine modules — a set of deliberately-scoped conventions, adapters, and shared packages that let modules interoperate without forcing premature unification. Where research found a genuine shared abstraction with two or more real v1 consumers (Arrow, OpenTelemetry GenAI conventions), LazyCore adopts it. Where research found that unification would be lossy or premature (export IRs, cross-module data contracts, LLM routing) — including `torch.export`/EXIR capture, which has only one real v1 consumer (LazyVision) since LazyEdge is deferred — LazyCore explicitly declines to build it and documents why; that capture step lives in LazyVision alone until a second real consumer exists.

### 2.1 Data/Tensor Foundation

LazyCore standardizes on **Apache Arrow as the canonical in-memory columnar format**, with a three-tier model reflecting where zero-copy interop is actually achievable versus where a real conversion step is unavoidable.

| Tier | Data class | Canonical format | Zero-copy interop | Consuming modules |
|---|---|---|---|---|
| 1 | Dense tabular / text / time-series | Arrow (pandas 2.x `ArrowDtype` backend, Polars) | Yes — pandas 2.x and Polars are interchangeable front-ends over the same Arrow buffers, confirmed near-zero-cost | AutoML module, LazyClean (pre-embedding stage), LazyForecast, LazyTune (metrics/configs), LazyRed (text payloads) |
| 2 | Sparse graph tensors | Dedicated sparse-tensor adapter (COO / CSR-CSC / SciPy sparse bridge, modeled on NVIDIA's "Universal Sparse Tensor" concept) | No — DLPack's spec is explicitly limited to dense/strided arrays and cannot represent sparsity; this is a genuine, irreducible conversion step | LazyGraph (bridging PyTorch Geometric's native COO and DGL's native CSR/CSC) |
| 3 | Dense image / audio | Native FFCV-style decode+augment pipeline, DLPack handoff only at the final dense-tensor stage | Partial — DLPack is used for the PyTorch handoff after decode/augment, but not for the decode stage itself, because image/audio workloads are bottlenecked by decode+augmentation *compute*, not by data-copy/serialization overhead (this is why FFCV outperforms both the PyTorch DataLoader and NVIDIA DALI) | LazyVision |

Two additional interop notes carry forward from module-specific research and apply platform-wide:

- **ONNX Runtime** (used by LazyClean specifically to stay under a 100MB footprint by avoiding a PyTorch dependency) supports DLPack-based zero-copy exchange with PyTorch/Arrow in the common case, but the boundary is less mature than PyTorch↔JAX↔CuPy interop — there are known, GitHub-issue-documented cases where CUDA-tensor roundtrips force CPU copies. Treat this as "mostly interoperable, minor known friction," not a blocking incompatibility.
- **Agent trajectories** (LazyAgent) are semi-structured/nested JSON. Flattening them to Arrow struct/list types for shared storage costs one conversion step. This is accepted as expected overhead, not a design flaw.

### 2.2 License Isolation Policy ("LazyIsolate")

LazyIsolate is a **deliberately-designed internal policy**, not an adopted external standard — research confirmed no industry-standard mitigation framework exists for this problem; SPDX and the Linux Foundation's SBOM tooling standardize license *identification*, not mitigation strategy.

| Risk Type | Required Mitigation | Notes |
|---|---|---|
| GPL/copyleft code needed at build/link time (e.g., SuiteSparse's CHOLMOD, GPLv2+) | Exclude via build flags/config; never compile/link against it (e.g., `CHOLMOD_CONFIG=-DNPARTITION`) | Solid, uncontested |
| Restrictively-licensed optional build dependency (e.g., METIS, University of Minnesota license) | Never enable the optional flag; route equivalent functionality through a permissively-licensed alternative (e.g., torch-cluster's Graclus algorithm instead of torch-sparse+METIS) | Solid |
| AGPL/GPL code needed at runtime, for internal-only local processing (never exposed as its own network-facing service to third parties) | Isolate via subprocess/IPC with a loosely-coupled, documented interface (CLI/pipe/socket) | Supported by the FSF's own "mere aggregation" framing and AGPL §13's focus on the covered program itself being network-exposed — but this is not a bright-line legal guarantee. **Flag for counsel.** |
| AGPL/GPL code that would itself be the network-facing/exposed service | Full replacement with a permissive alternative, or explicit dual-licensing negotiation | Subprocess-isolation theory does not apply here; high risk |
| Source-available non-compete license (e.g., FSL-1.1-MIT, BUSL) covering code you want equivalent functionality of | Clean-room reimplementation, rigorously documented (no engineer with prior access to the licensed source participates; spec-only reference) | Legally sound because you are never a licensee of the covered code — the Competing Use clause restricts use of the licensed software itself, not independent reimplementation. **Flag for counsel** to formalize the clean-room protocol. |
| Non-commercial-licensed weights/data (e.g., CC BY-NC 4.0, such as Salesforce's MOIRAI) | Isolate into a separate, opt-in-gated plugin package; load only if the user explicitly sets an acceptance flag (e.g., `accept_restricted_licenses=True`) | Standard approach; corresponds to Tier 2 of the Model Licensing Allowlist Policy (§2.10) |

### 2.3 Sandbox Execution (shared between LazyRed and LazyAgent)

LazyCore provides **one shared sandbox executor and adapter base class**, with mode-specific policy configs layered on top per module — not fully separate implementations. This is viable because both modules share the same underlying threat class at the kernel level: containing arbitrary code execution. LazyAgent's threat model is a benchmarked agent generating and running arbitrary shell/code; LazyRed's kernel-level threat is the same class of concern when a red-team target executes tool calls. LazyRed's *other* threats — prompt-injection-driven credential leakage, unauthorized-but-legitimate-looking tool calls — are semantic/policy-level, not kernel-level, and are handled separately (§2.4).

**Correction to prior draft research:** "Aegis" was previously miscited as a sandboxing tool. No sandboxing tool exists under that name. Real hits for "Aegis" in this space (e.g., NVIDIA's Aegis content-safety dataset, various LLM API security gateway/proxy products) are guardrail/firewall products and belong in §2.4, not in the sandbox executor evaluation.

Candidate sandbox technologies identified by prior research — Docker/runc, gVisor/runsc (~50ms cold start via user-space kernel interception), Firecracker microVMs (~125ms cold start, hardware KVM), OpenSandbox (Alibaba, Docker/Kubernetes-based), CubeSandbox (Tencent Cloud, per-sandbox microVM via KVM), and the lightweight namespace-based "SWE-MiniSandbox" pattern — are **all fundamentally Linux-native**. None has a macOS equivalent, and this platform's reference machine and primary v1 target is Apple Silicon macOS.

#### 2.3.1 The Mac GPU-in-sandbox problem (2026 findings)

Fresh research into this specific gap confirms the concern raised by prior Linux-only sandboxing research: **Docker Desktop and Podman on macOS still run containers inside a hidden Linux virtual machine, and that VM boundary has not historically exposed Metal/MPS to containerized PyTorch processes** — MPS acceleration inside a standard Docker container on Mac is not natively supported ([Stack Overflow](https://stackoverflow.com/questions/79541677/how-to-enable-mps-acceleration-for-pytorch-inside-docker-on-mac)).

There has been real movement in 2025–2026, but it doesn't close the gap cleanly:

- **Apple's `container`/Containerization framework** (open-sourced at WWDC25, Swift-based, built on the Virtualization framework, described by observers as effectively "Kata Containers for macOS" — [Edera](https://edera.dev/stories/apple-just-validated-hypervisor-isolated-containers-heres-what-that-means), [Apple open source](https://opensource.apple.com/projects/containerization)) runs Linux containers each in its own lightweight per-container VM. As of the active GitHub discussion on this exact question, **GPU passthrough is still unresolved/requested, not shipped** ([apple/container Discussion #62](https://github.com/apple/container/discussions/62)).
- **Podman + libkrun** and **Vulkan-remoting** approaches (demonstrated at DevConf.US 2024, covered by [Andreas Kunar](https://medium.com/@andreask_75652/gpu-accelerated-containers-for-m1-m2-m3-macs-237556e5fe0b)) can get GPU-accelerated compute into a container via Vulkan translation, but this is a narrow, non-default, community-maintained path, not a first-class Metal/MPS passthrough for PyTorch.
- **Docker Model Runner's new `vllm-metal` backend** ([Docker blog](https://www.docker.com/blog/docker-model-runner-vllm-metal-macos)) is the closest thing to "GPU inference alongside Docker on Mac" shipped in 2026 — but it works by running the MLX/Metal inference process **on the host**, outside the container's Linux VM boundary, and exposing it to containers via an API. It is not GPU passthrough *into* the isolation boundary.

On the native-macOS-primitive side, **`sandbox-exec`/Seatbelt** (Apple's syscall/file/network allowlist sandbox, deprecated-but-functional, used in production by agent harnesses including Gemini CLI and several open-source "agent-in-a-sandbox" projects — [Pierce Freeman's survey](https://pierce.dev/notes/a-deep-dive-on-agent-sandboxes), [michaelneale/agent-seatbelt-sandbox](https://github.com/michaelneale/agent-seatbelt-sandbox)) is confirmed to be a real, actively-used isolation primitive for constraining untrusted *tool-calling* processes on macOS. Critically, research also surfaced a directly relevant technical fact: **Seatbelt cannot block or gate Metal/GPU/Cocoa access even if you wanted it to** — "GPU and display passthrough flags have no effect on macOS because Metal and Cocoa are system-level and cannot be blocked via SBPL" ([akitaonrails/ai-jail DeepWiki](https://deepwiki.com/akitaonrails/ai-jail/4.5-macos:-seatbelt-sandboxing)). Seatbelt is a process-level allowlist, not a VM/kernel boundary — it restricts filesystem, network, and syscall surface without needing (or being able) to mediate the GPU driver stack at all.

**Recommendation for v1:** Do not attempt to run the GPU-bound model inference process itself inside a hard isolation boundary on Mac. Instead, adopt a **split-trust architecture**:

1. **Inference (trusted, GPU-bound):** the loaded local model (MLX/PyTorch-MPS) runs unsandboxed, in-process or in a lightly-isolated host process. This is safe because the actual risk surface is local model weights and forward-pass compute, not adversarial code — there is nothing here for a kernel-level sandbox to usefully contain.
2. **Tool-calling / code-execution layer (untrusted, CPU-bound):** the agent's generated shell commands, code execution, and file/network access — the actual adversarial surface in both LazyRed's and LazyAgent's threat models — is what gets sandboxed, via the shared executor from this section. On Mac, that means `sandbox-exec`/Seatbelt (or Apple's `container`/Containerization framework once/if GPU passthrough is irrelevant to that layer, since it's CPU-only anyway) as the v1 default; on Linux (secondary v1 target), the existing gVisor/Firecracker/namespace options from prior research apply unchanged, since Linux has no VM-boundary GPU problem to begin with.

This means the shared sandbox executor's job is never to contain a GPU process — GPU/MPS access is deliberately kept outside the isolation boundary on both platforms, and the executor's adapter layer only needs a Mac-specific backend (Seatbelt) and a Linux-specific backend (gVisor/Firecracker/namespaces) for the one thing that's actually shared: constraining the orchestration/tool-execution layer.

### 2.4 Guardrail/Firewall Layer (LazyRed-specific)

A distinct subsystem from the sandbox executor in §2.3, responsible for detecting and blocking prompt injection, PII/credential leakage, and policy violations at the prompt-response *semantic* level — the "Aegis-category" of tools/products correctly relocated here per the correction in §2.3. This layer is core to LazyRed's red-teaming threat model. It is explicitly **not** needed by LazyAgent's benchmark-eval use case, which only cares about task success and kernel-level containment, not semantic policy enforcement.

### 2.5 Export/Compilation Architecture

Three top-level backends remain **permanently separate**, not unified through one shared IR — ONNX cannot represent LLM-serving-specific concerns (KV-cache, paged attention, quantization kernels) or edge/microcontroller-specific concerns (static memory planning).

| Backend | Scope | Modules | v1 status |
|---|---|---|---|
| 1 — ONNX / ONNX Runtime | Shared terminal execution format for non-edge, non-LLM-serving targets | AutoML (via skl2onnx), LazyVision (via `torch.export` → onnx-graphsurgeon) | In scope |
| 2 — LLM local serving/quantization | **Narrowed for v1** to local-only formats: GGUF (llama.cpp) and MLX-native (Apple Silicon, matching MPS as primary v1 backend) | LazyTune | In scope, narrowed — vLLM/SGLang/TensorRT-LLM/AutoAWQ-Marlin are cloud/datacenter-GPU-serving concerns, explicitly out of scope for v1's local-only mandate, deferred to a later cloud-enabled phase alongside LazyRed/LazyAgent's cloud targets |
| 3 — Edge/microcontroller compilation | Entirely deferred | LazyEdge | Out of scope for v1; roadmap only |

`torch.export()`/EXIR would legitimately be a **shared frontend/capture step** between LazyVision and a future edge-compilation module, up through common ATen-dialect decomposition passes, if both existed today. They don't: LazyEdge is fully deferred (Part 6), so this capture step has exactly one real v1 consumer (LazyVision) and lives there — not in LazyCore, per §2.9's "don't build shared infrastructure before two real consumers need it" rule. Revisit extracting a shared capture layer only once LazyEdge is actually un-deferred and genuinely needs it.

### 2.6 Telemetry/Tracing Schema

LazyCore adopts **OpenTelemetry GenAI semantic conventions** as the shared trace/result schema across LazyRed's security-audit reports, LazyAgent's execution trajectories, and the leaderboard-style results from the AutoML module, LazyForecast, LazyGraph, and LazyVision. Data is represented as OTel spans plus custom attributes (e.g., `security.severity`, `owasp.mapping`, `ml.metric.accuracy`) and span events for conversational transcripts. MLflow has already standardized on this convention, making this a genuine "adopt an existing standard" decision rather than a custom schema design.

### 2.7 Packaging Model

**[HISTORICAL — ORIGINAL DECISION, SUPERSEDED BELOW]**

LazyCore follows a **Hugging Face-style model of independently-versioned, separate packages per module**, sharing one thin `lazycore` package for common schemas/interfaces with near-zero dependencies — explicitly **not** one monorepo with pip extras. PyTorch-heavy modules (LazyTune, LazyVision, LazyGraph), the PyTorch-free LazyClean, and the Node-adjacent tooling implied by LazyRed's Promptfoo integration have genuinely conflicting dependency universes that pip's extras resolver cannot cleanly reconcile.

**[HISTORICAL — REFINEMENT, ALSO SUPERSEDED BELOW]**

**Refinement (added post-v1, at explicit user request):** a thin, additive umbrella package, `benchcraft-ml` (`packages/benchcraft-ml`), MAY unify the nine modules under one top-level import namespace (`import benchcraft_ml; benchcraft_ml.automl.compile(...)`), provided (a) it has zero hard dependencies beyond `lazycore`/`benchcraft-core`, (b) every module's real dependency stack remains strictly opt-in via its own named extra (e.g. `pip install benchcraft-ml[tune]`), so no environment is ever forced to install two conflicting dependency stacks together, and (c) the umbrella package contains no reimplemented logic — only re-export shims pointing at the already-independently-versioned standalone packages (see `packages/benchcraft-ml/README.md`). The nine standalone packages remain independently published and installable exactly as before; the umbrella package is a convenience layer on top, not a replacement. This refines, but does not reverse, the original "don't unify under one dependency tree or pip-extras setup" decision above — the thing that decision was protecting against (forced co-installation of conflicting dependency stacks) still cannot happen; only the *import ergonomics* have been unified, not the dependency resolution.

*(Note: `benchcraft-ml` was never published or merged — it was abandoned before release, superseded by the reversal below. Its name is left as originally written here, rather than mechanically renamed to a `dscraft`-branded equivalent, because it refers to a specific, literal package/import identifier that was actually proposed under that name at the time; renaming it would misrepresent what was actually decided historically. See the reversal immediately below for what replaced it.)*

**Reversal (superseding the above, at explicit user request, 2026-07):** Both the original per-module independent-packaging decision and the subsequent `benchcraft-ml` thin-wrapper refinement are now **superseded**. Neither is in effect going forward.

The new model: **one real package, `dscraft`**, with real code — not shims, not re-export wrappers — living in purpose-based subpackages (`dscraft.core`, `dscraft.automl`, `dscraft.clean`, `dscraft.forecast`, `dscraft.graph`, `dscraft.vision`, `dscraft.tune`, `dscraft.security`, `dscraft.agent`). Each subpackage's heavy dependencies are gated behind its own optional extra (`pip install dscraft[automl]`, `pip install dscraft[vision]`, etc.), so installing one subpackage never forces installing another's conflicting dependency stack. This preserves the *underlying concern* the original §2.7 decision protected against — no environment is ever forced to install the PyTorch-heavy, PyTorch-free, and Node-adjacent dependency stacks all at once — while changing the *distribution* model to match established industry practice for exactly this shape of problem: one core package with real, importable subpackages, each gating its own heavy dependencies behind an extra (e.g. `numpy` → `numpy.linalg`, `pytorch` → `torch.nn`/`torchvision`), rather than nine separately-published sibling packages plus a thin umbrella shimming over them.

Concretely, this means:

- There is exactly one PyPI distribution (`dscraft`), not nine (plus a since-abandoned tenth umbrella).
- `import dscraft.automl` (etc.) is real code executing in that subpackage's own module tree, not a re-export shim pointing at a separately-versioned sibling package — the `benchcraft-ml` refinement's re-export-only constraint no longer applies because there is nothing left to re-export *from*.
- The extras mechanism (`pip install dscraft[automl]`, `dscraft[all]`) is the *only* distribution-time dependency-isolation lever now — there is no longer a second, separate umbrella-package layer with its own extras on top of nine base packages' own extras.
- `dscraft.core` (the direct successor to `lazycore`) remains the one required base dependency, exactly as `lazycore` was before.

See `packages/dscraft/pyproject.toml` for the actual extras layout and `packages/dscraft/README.md` for the full up-to-date package description; this section documents the *decision history*, not the current implementation detail, which lives with the code.

### 2.8 Target/Connection Abstraction

**No LLM router, no multi-provider abstraction, no LiteLLM dependency** — explicitly descoped per stakeholder directive: this platform is "not an LLM router in any way, not part of MVP or even the first few versions." v1 uses a bare-minimum "bring your own connection" model: each module (LazyTune, LazyRed, LazyAgent) defines its own minimal, module-specific way to reference a local model handle (a loaded HF/PyTorch/MLX model object, or a local model file path). No shared `BaseTarget` interface is being formally designed yet — that is deferred until a concrete cross-module need emerges. Remote/API endpoints are not a v1 concern.

### 2.9 Interface Contracts Between Modules

Formal inter-module data contracts are explicitly deferred as too early — no two modules yet have a concrete need to hand data to each other. When that need arises, research has already identified the right pattern to adopt: **MLflow's Model "flavors" pattern** — a typed manifest plus signature plus storage URI, with graceful fallback to a generic format. This is noted here as the future direction, not designed now.

### 2.10 Model Licensing Allowlist Policy

Each module maintains an explicit, per-module allowlist of pre-approved, commercially-safe-by-default model checkpoints, split into two tiers:

- **Tier 1** — permissive license (e.g., Apache-2.0, MIT): auto-usable by default.
- **Tier 2** — restricted license (e.g., CC BY-NC): isolated behind an explicit opt-in acceptance flag, per the mitigation defined in §2.2's license isolation table.

This extends the pattern already validated in prior LazyForecast-specific research — Apache-2.0 TimesFM and Chronos-Bolt as Tier 1 defaults, CC BY-NC MOIRAI as Tier 2 opt-in — to the whole platform. Populating and maintaining the actual per-model allowlists is an ongoing maintenance task for each module owner, not a one-time design exercise.

A read-only audit of three of the stakeholder's other active projects (`himalaya_email_classifier`, `hermes-rag-qdrant-efficient-skill`, `ai-helpdesk-agent`) was run to check which models the stakeholder already relies on regularly, so those choices are reflected here rather than rediscovered later. The audit surfaced two categories of finding, handled differently per stakeholder direction:

This validates and should elevate LazyRed's "Multi-Model Jury Consensus" mitigation (currently Appendix A only) into its main Part 3 description, since it's a pattern already proven useful in the stakeholder's real workloads, not just a research idea.

**Hosted-API LLMs — explicitly out of v1 scope, documented not resolved**: the stakeholder's actual production judge/generation models across all three audited projects are hosted APIs — Claude (Sonnet/Haiku tiers), Gemini (2.5 Flash/Pro, `gemini-embedding-001`), and Ollama *Cloud* (the `:cloud`-suffixed models, e.g. `glm-5.2:cloud`, `kimi-k2.7-code:cloud`, `minimax-m3:cloud` — these are hosted, not local weights, despite running through the Ollama CLI). This does not fit DSCraft v1's local-only, no-router, bring-your-own-local-model-handle boundary (§2.8), and the stakeholder has confirmed this gap should stay unresolved by design: v1 does not attempt to absorb hosted-API judge/generation workflows, and this exclusion should be documented (here, and in any future roadmap discussion) as a known, deliberate boundary rather than a missing feature.

## Part 3: Per-Module Architecture Summaries

The nine module subsections below each condense prior deep-research findings and apply the platform-wide v1 constraints established above: local-only execution (no cloud/remote-serving targets), Apple Silicon M4 Max/128GB-unified-memory as the primary reference hardware with MPS as the primary backend (CUDA secondary), no LLM router or multi-provider abstraction, MIT-only platform code with a Tier 1/Tier 2 allowlist for external checkpoints, independent per-module packaging over one shared core, and OpenTelemetry GenAI semantic conventions for all result telemetry.

### Module 1: AutoML — STATUS: IN v1

A clean-room, LazyPredict-successor tabular AutoML engine positioned against LazyPredict (fragile API, OOMs at scale), PyCaret (combinatorial-explosion OOM risk, namespace collisions), FLAML (unmanaged GPU OOM, class-vanishing under imbalanced downsampling), MLJAR (disk-I/O latency from heavy reporting), and AutoGluon (resource-heavy, stacked-overfitting-prone). Core dependency surface is minimal (numpy/pandas/scikit-learn only), with boosting libraries, SHAP explainability, and ONNX export gated as lazy-loaded optional extras. Three signature capabilities: streaming/incremental optimization via `partial_fit` with a fading-factor running-metric evaluator; zero-config drift detection using PSI computed on a non-blocking background thread to auto-trigger retraining alerts; and a `.compile()` path that fuses a full `sklearn.pipeline.Pipeline` into a single ONNX graph via skl2onnx, replacing brittle pickle-based serialization. Licensing is the critical constraint: PyCaret 4.0's core moved to a split FSL-1.1-MIT/BUSL-1.1 license with a non-compete clause and delayed 2-year MIT rollover, so direct reuse is prohibited and the module must be an independent reimplementation of the API surface only. TPOT (GPL-3.0) and its DEAP dependency (LGPL) are excluded from the dependency tree entirely. No v1 rescoping was needed beyond standard constraint application — the design was already local/CPU-first and license-clean by construction.

### Module 2: LazyClean — STATUS: IN v1

A unified data-quality firewall combining Decoupled Confident Learning (DeCoLe) for tabular label-error detection — using per-subpopulation confidence thresholds instead of one global threshold, avoiding the standard confident-learning failure mode of over-pruning valid minority-class samples — with Density-Based Semantic Deduplication (D4) for text/embedding near-duplicate pruning via spherical mini-batch k-means plus IVF-HNSW approximate nearest-neighbor indexing, avoiding O(n²) pairwise cosine-similarity cost at scale. Embeddings are generated via native ONNX Runtime rather than PyTorch, deliberately keeping the runtime dependency footprint under 100MB and avoiding the PyTorch/HuggingFace stack. Additional capabilities include automated train/test contamination auditing (IVF cosine-similarity index of training embeddings queried against test embeddings) and a single aggregate "Dataset Integrity Score" combining structural (PII/corruption/readability), label-auditing, and deduplication sub-scores, surfaced as a row-level audit leaderboard mapping issues to exact dataframe indices. The dependency stack is fully MIT throughout, a deliberate differentiator against Cleanlab (post-acquisition licensing uncertainty), Snorkel (shifted to proprietary Snorkel Flow), Lilac (scale features gated behind hosted Lilac Garden), and Great Expectations (orchestration requiring GX Cloud). Architecture is local-first and lightweight by design, requiring no rescoping for v1 constraints.

### Module 3: LazyForecast — STATUS: IN v1

Reconciles three forecasting paradigms — classical statistical models (AutoARIMA/AutoETS via Nixtla StatsForecast), tree-based ML (LightGBM/XGBoost via Nixtla MLForecast), and zero-shot Time Series Foundation Models (TimesFM 2.5, Chronos-Bolt, Lag-Llama, PatchTST) — under a single Polars/Arrow zero-copy pipeline with two branches: a tabularization branch (SIMD-vectorized rolling-window/lag features for classical/ML models) and a direct tensor-export branch (DLPack/IPC zero-copy handoff to PyTorch/JAX for TSFMs). A self-healing preprocessing engine normalizes timezones to UTC, vectorized-upsamples irregular series, and applies paradigm-aware imputation (linear interpolation for classical models, dedicated mask tokens for neural/TSFM patch embeddings). Uncertainty quantification uses two conformal-prediction protocols — MSCP (per-horizon-step quantile thresholds, for classical/TSFM models) and EnbPI (leave-one-out bootstrap residuals, for tree ensembles) — scored on a unified leaderboard via PICP, MPIW, and the Gneiting-Raftery interval score. Key licensing finding: Salesforce MOIRAI's code is Apache-2.0 but its weights are CC BY-NC 4.0, so it is isolated as a Tier-2 opt-in-gated plugin; the default TSFM bench instead standardizes on fully-Apache-2.0 TimesFM 2.5 and Chronos-Bolt. Given the 128GB unified-memory reference hardware and MPS-primary target, TSFM inference runs comfortably local with no cloud GPU dependency — this module required no scope reduction, only confirmation that its neural tier fits the local-only constraint.

### Module 4: LazyGraph — STATUS: IN v1

Unifies neighborhood-aggregation Message-Passing Neural Networks (GCN's fixed-topology aggregation fusable into sparse-dense matmul; GAT's anisotropic attention requiring edge-parallel scatter-reduction; GraphSAGE's decoupled bipartite aggregation) and Global Graph Transformers (Laplacian Positional Encodings injecting topological awareness into permutation-equivariant attention) across PyTorch Geometric (COO-native) and DGL (CSR/CSC-native) via the Universal Sparse Tensor layer's dedicated COO/CSR-CSC adapter (§2.1 Tier 2) — per §2.1, DLPack cannot represent sparsity, so this is a real conversion step, not a zero-copy handoff; `__cuda_array_interface__`/DLPack only re-enter once a tensor is dense (e.g. node feature matrices passed to a GNN layer). Includes an automated relational-database-to-graph mapping engine (SQL schema reflection, FK-to-edge conversion, semantic-type-aware column featurization, strict temporal filtering to prevent leakage) and addresses the neighborhood-explosion problem via three sampling paradigms — node-wise (GraphSAGE-style), layer-wise importance (FastGCN-style), and layer-dependent importance sampling (LADIES, the most balanced) — plus training-pathology monitoring via Dirichlet-Energy/Rayleigh-Coefficient tracking (oversmoothing) and Jacobian-sensitivity/discrete-curvature analysis (oversquashing). Licensing requires two build-time exclusions: METIS (restrictive University of Minnesota academic license, an optional torch-sparse flag), routed instead through torch-cluster's permissively-licensed Graclus algorithm, and CHOLMOD/SuiteSparse (GPLv2+), routed through SciPy/LAPACK on CPU or NVIDIA cuSOLVER on GPU. Both exclusions are enforced at build time rather than as runtime workarounds. No further v1 rescoping needed; the architecture is already local-execution-oriented.

### Module 5: LazyVision — STATUS: IN v1 (MPS-targeted)

Unifies CNN classifiers, Vision Transformers, real-time object detectors (YOLO-family, D-FINE's probability-distribution box regression, RT-DETR's NMS-free set prediction via learned object queries), and acoustic/spectrogram models (Mel-spectrogram STFT preprocessing) under one preprocessing abstraction, backed by a native Rust/PyO3 data-loading layer with DLPack zero-copy handoff to eliminate both multiprocessing IPC/GIL serialization overhead and the copy-on-read problem (Python reference-counting forcing silent per-worker duplication of shared-memory dataset metadata). Overfitting/instability on small custom ViT datasets is addressed via Sharpness-Aware Minimization, Layer-wise Learning Rate Decay, and hybrid local-convolutional spatial-locality modules; inference-export failures are addressed via `torch.export`/TorchDynamo symbolic shape tracing and `onnx-graphsurgeon` custom-operator mapping (e.g., NMS). Licensing requires isolating AGPL-3.0 detectors (Ultralytics YOLOv8/v11, Tsinghua YOLOv12) and GPL-3.0 RTMDet into a dynamic plugin architecture running in separate subprocesses/virtualenvs (treated as "mere aggregation" per FSF guidance, flagged for counsel review), with Apache-2.0/MIT alternatives (D-FINE, RT-DETR, YOLOX, LibreYOLO) as the non-copyleft default. **v1-specific work item**: since MPS is the primary backend rather than CUDA, the team must audit MPS support maturity for attention kernels and NMS layers before finalizing the initial supported-model list — models requiring CPU fallback should be flagged rather than assumed to have CUDA parity.

### Module 6: LazyTune — STATUS: IN v1, LOCAL MODELS ONLY

Unifies Axolotl (declarative YAML over DeepSpeed/FSDP), LLaMA-Factory (broad compatibility, fragile custom-kernel integration, RCE risk from user-controlled model paths), Unsloth (hand-optimized Triton kernels, up to 70% memory reduction/2-5x speedup, GRPO "Standby" weight-sharing with vLLM), Hugging Face TRL (foundational but unoptimized), and torchtune (composable recipes with in-backward optimizer fusion and Linear Cross-Entropy loss avoiding full vocab-logit materialization) under an Adapter-Factory pattern: a `BaseTrainingAdapter` with in-process `ProgrammaticAdapter`s (Unsloth, TRL) and subprocess-isolated `SubprocessAdapter`s (torchtune, Axolotl via `torchrun`). A multi-fidelity Micro-Tuning system trains small proxy models on a 1% data subsample via BOHB across parallel configs, then extrapolates full-scale convergence via Multi-Power-Law fitting and Shifted-Power-Law scale translation, claimed to cut hyperparameter search cost ~90%. Catastrophic forgetting and RL reward-hacking are mitigated via KL-penalization against a frozen reference policy, blended anchor datasets, and multi-objective reward shaping. **v1 rescope**: the original vLLM/SGLang/TensorRT-LLM cloud-serving export pipeline and AutoAWQ/Marlin quantization are entirely out of scope; export narrows to local-serving formats only — GGUF (llama.cpp) and MLX-native (Apple Silicon-optimized) — with CUDA-quantization-kernel work deferred to a future cloud-enabled phase. Default model-selection guidance must favor MoE over dense architectures at large parameter counts (e.g., 70B-total/~3B-active MoE is genuinely fast on 128GB unified memory; dense 70B quantized is not).

### Module 7: LazyRed — STATUS: IN v1, LOCAL TARGETS ONLY (cloud targets deferred)

Unifies garak (Probes/Detectors/Generators/Evaluators/Harnesses/Buffs around an `attempt` transaction object), DeepTeam (simulation-led payload generation with single-turn enhancements and multi-turn progressions like Crescendo/Tree-Jailbreaking, LLM-as-Judge G-Eval scoring), PyRIT (multi-agent orchestration with persistent DuckDB/SQLite conversational memory and chained Converters), and Promptfoo Red Team (CI/CD-integrated Plugin/Strategy/Target architecture) under a `BaseSecurityAdapter` pattern, all Apache-2.0/MIT licensed. Runs fully offline-first: default telemetry disabled per tool (`PROMPTFOO_DISABLE_TELEMETRY`, `DEEPEVAL_TELEMETRY_OPT_OUT`, local-only PyRIT memory instead of Azure SQL), with mutation/grading routed to local inference. An Adversarial Target-Mutator Layer uses the "TopicAttack" paradigm (contextual topic-transition bridge plus targeted reminder-prompt) to test RAG indirect-prompt-injection resilience beyond easily-filtered blunt payloads. Findings compile into a unified JSONL report mapped to OWASP LLM Top 10 / OWASP Agentic Top 10 / MITRE ATLAS IDs on a Vulnerability & Failure Rate Leaderboard. Judge/scoring reliability follows a **Multi-Model Jury Consensus** pattern — a heterogeneous panel of judges deliberately drawn from non-overlapping model lineages (never two judges from the same vendor/training family) votes on each verdict, since a single judge model reliably reproduces its own family's blind spots; this was originally an Appendix A research detail but is elevated here because it's a pattern the stakeholder has already independently validated as necessary in production red-team/eval workloads outside this platform (see §2.10.1's cross-project audit). **Correction applied**: a prior draft miscited a nonexistent sandboxing tool "Aegis"; genuine Aegis-named products are AI-safety/guardrail tools and have been re-homed as this module's dedicated Guardrail/Firewall Layer, distinct from the sandbox executor now shared with LazyAgent (mode-specific policies, shared underlying executor/adapter base class). v1 scope is local-model targets only; remote/cloud targets are deferred.

### Module 8: LazyAgent — STATUS: IN v1, LOCAL TARGETS ONLY (cloud targets deferred)

Surveys Inspect AI (composable solver/scorer library with pluggable sandbox providers), hal-harness (centralized leaderboard runner across AppWorld/USACO/tau-bench/CORE-bench/SWE-bench Verified, but tightly coupled to W&B Weave — a SaaS lock-in this platform avoids), AutoRAG (sequential grid-search RAG optimization, prone to stage-cascade failures since early retrieval metrics don't reliably predict generation quality), native SWE-bench runners (heavyweight, up to 2TB nested Docker caches, x86_64-specific builds that struggle under ARM64 emulation), and MASEval (framework-agnostic `AgentAdapter` interface spanning smolagents/LangGraph/AutoGen/CAMEL). Implements a Bring-Your-Own-Agent universal adapter and a Multi-Objective Pareto RAG Optimization loop (accuracy vs. latency vs. cost) using DISCO-style sample condensation to extract a ~1% informative task subset, claimed to cut evaluation compute cost ~99% within a 2% performance-estimation margin. This module's sandboxing research (Docker/runc vs. gVisor's Sentry-kernel syscall interception vs. Firecracker microVMs vs. a lightweight SWE-MiniSandbox, with default-deny egress, read-only root filesystem, cgroup v2 limits, credential-scrubbing proxy) is the platform's most rigorous and forms the basis of the shared sandbox executor also used by LazyRed. **v1 rescope**: given ARM64/Apple-Silicon friction with Docker-based SWE-bench-style isolation, the initial benchmark suite should prioritize tasks compatible with the Mac-first sandbox strategy (§2.3.1) rather than assume Linux-native parity; remote/cloud-hosted agent targets are deferred.

### Module 9: LazyEdge — STATUS: DEFERRED (roadmap only, internal reference — not user-facing)

A cross-runtime, hardware-aware edge/microcontroller compiler unifying ExecuTorch (torch.export→EXIR→Core ATen→Edge/Backend Dialect→flatbuffer `.pte`, zero-heap-allocation C++ runtime), LiteRT/TFLite (OpResolver interpretation plus AOT NPU compilation for Qualcomm/Intel), OpenVINO (nGraph IR, host-CPU ISA-aware JIT codegen), Core ML (MIL-based compilation to `.mlpackage`/`.mlmodelc`, static shapes to Apple Neural Engine, dynamic shapes to GPU pipeline), and TVM/microTVM (Relay→Tensor IR with Unified Static Memory Planning for bare-metal targets) — bridged via a Virtual Runtime Bridge (`LazyEdgeRuntime` C++ interface with `dlopen`/`dlsym`-loaded target-specific shims to avoid binary bloat and dependency conflicts). Its 5-phase pipeline covers capture/lowering, hardware partitioning with constant-folding, INT4/INT8 quantization with overflow-aware dyadic scaling, static memory planning via liveness analysis, and C/C++ codegen with Execute-in-Place flash-resident weights, plus NSGA-II/Bayesian multi-objective Pareto optimization against hardware-in-the-loop power/latency/memory profiling. Core runtime licenses are clean (BSD-3-Clause ExecuTorch; Apache-2.0 LiteRT/OpenVINO/TVM/CMSIS-NN), with the main risk being proprietary vendor SDKs (Qualcomm QNN, Apple `coreai-build`/coremltools binaries) requiring detection-and-link isolation rather than bundling. **Deferral rationale**: this module targets compiling trained models for embedded/edge hardware distinct from the local development workstation — fundamentally a production/deployment concern outside v1's local-development-only scope. The architecture is retained here as an internal roadmap reference only and is not exposed in any user-facing v1 documentation.

## Part 5: Legal Review Appendix

This appendix consolidates every item in the DSCraft architecture that requires actual counsel review before shipping or relying on at business scale. None of these are resolved by engineering decisions alone — they are flagged here specifically because the engineering mitigations already in place reduce risk but do not substitute for legal sign-off.

**1. AGPL/GPL-at-runtime subprocess isolation strategy (LazyVision AGPL-licensed detector models)**

- **Context**: LazyVision's design uses subprocess isolation to run certain AGPL-licensed detector models as a separately-invoked process rather than linked in-process, on the reasoning (borrowed from FSF's own "mere aggregation" language) that separately-executing programs communicating via inputs/outputs are not necessarily a "combined work" for copyleft purposes.
- **What needs counsel review**: Whether the platform's specific subprocess isolation implementation (invocation method, data interchange format, process boundary, packaging/distribution unit) actually qualifies as "mere aggregation" under AGPL/GPL, for the platform's actual intended distribution and internal-use pattern. This is architecture-specific — no generic answer applies.
- **Flag**: "Mere aggregation" is FSF-endorsed interpretive reasoning, not a bright-line judicial guarantee. No case law establishes a precise technical threshold (e.g., a specific IPC mechanism) that is definitively safe. Treat this as a risk-reduction posture, not a cleared position, until reviewed.
- **Mitigation already in place**: Detector models isolated in a separate subprocess/process boundary rather than in-process linkage; no shared memory space or compiled linkage with AGPL code.

**2. Clean-room reimplementation protocol for source-available non-compete licenses (FSL-1.1-MIT, AutoML module's PyCaret-equivalent functionality)**

- **Context**: The AutoML module's PyCaret-equivalent functionality is being built against/inspired by source-available code under a license (FSL-1.1-MIT-style) that includes non-compete terms, converting to a permissive license only after a delay period.
- **What needs counsel review**: Whether the informal internal understanding of "we didn't copy code, we reimplemented the approach" is legally sufficient, or whether a formally documented clean-room process (contamination-free specification, a "no prior access" implementation team separate from anyone who reviewed the licensed source, documented provenance of all authored code) is required to make the non-compete/independent-creation defense credible.
- **Flag**: An informal, undocumented claim of independent reimplementation is not the same thing as a defensible clean-room record. If this is ever challenged, the absence of contemporaneous documentation is itself a liability.
- **Mitigation already in place**: None yet formalized — this item currently rests on team practice, not a documented process. This is the single highest-priority item in this appendix precisely because no paper trail currently exists.

**3. Tier 1 / Tier 2 model licensing allowlist — ongoing re-verification**

- **Context**: External models used or recommended by the platform are classified Tier 1 (permissive, business-safe, auto-usable) or Tier 2 (restricted, opt-in-gated behind an explicit acceptance flag), maintained per module.
- **What needs counsel review**: Establish a recurring (not one-time) legal review cadence for the allowlist, since model checkpoints and their associated licenses can change after initial classification (license terms amended upstream, a checkpoint re-released under different terms, or an initial classification simply being wrong).
- **Flag**: This is a maintenance/process risk, not a point-in-time risk — the allowlist can silently go stale as the model ecosystem moves.
- **Mitigation already in place**: The Tier 1/Tier 2 structure itself, and the opt-in acceptance-flag gate for Tier 2 models, provide a mechanical safeguard against accidental Tier 2 use, but do not substitute for periodic re-verification of classifications.

**4. License Isolation Policy — internal framework, not an industry standard**

- **Context**: A License Isolation Policy decision table (§2.2) sets out per-module strategies for isolating differently-licensed dependencies and models.
- **What needs counsel review**: The table should be reviewed by counsel as a whole, as a single artifact, before the business relies on it at scale.
- **Flag**: This framework was deliberately designed for DSCraft's specific situation; it is not an adopted industry standard, and no such standard currently exists for this exact problem (mixed-license ML tooling isolation). Do not treat its existence as itself a form of legal validation.
- **Mitigation already in place**: The framework was constructed from FSF guidance and observed practice in comparable OSS ML projects, but this is a design input, not a substitute for review.

## Part 6: Open Items / Internal Roadmap

**This section is internal-only.** It is for the stakeholder's own future planning and must not be exposed in end-user-facing documentation, marketing, or scope descriptions. Everything below was deliberately excluded from v1, with reasoning, not simply left undone.

- **An MCP server and a Skill, so Hermes and other AI agent tools can consume DSCraft as callable tools.** DSCraft is intended to eventually expose itself as an MCP server plus an installable Skill, so AI agents (Hermes among them, alongside any other MCP-capable tool) can call DSCraft's modules as tools, rather than DSCraft remaining a Python-only library that only a human developer imports directly. This is a generic capability of the library, not a bespoke integration for one consumer. **Explicitly confirmed not a v1 requirement** — v1 stays a plain pip-installable library per Part 1/§2.7's packaging model; the MCP server and Skill are a post-v1 consumption layer built on top of the stable per-module public APIs once those APIs are proven out. No module's v1 API should be shaped around this prematurely (consistent with §2.9's "don't build formal interface contracts before a concrete need exists" reasoning), but package boundaries and public APIs should stay clean enough that wrapping them in an MCP server later doesn't require a breaking rework. Revisit once there's a concrete consumption pattern to design the MCP tool surface against.

- **LazyEdge (edge/microcontroller compilation) — full deferred module.** Originally one of the nine researched modules. Deferred in its entirety because it is fundamentally a production/cloud-deployment concern (compiling models for constrained embedded targets), which doesn't fit a local-dev-workstation-focused v1. Revisit only if/when the platform's scope intentionally expands toward deployment tooling.

- **Cloud/remote target support for LazyRed and LazyAgent.** Both modules are scoped to local targets only in v1 (local models for red-teaming, local agents/RAG pipelines for benchmark eval). Cloud target support (testing/evaluating remotely-hosted models or agents) is a natural extension but was excluded to keep v1 consistent with the platform's local-first positioning. Revisit once local-target functionality is mature.

- **LLM router / multi-provider target abstraction.** Explicitly and deliberately not pursued — the stakeholder does not want this pursued casually or by default. This should only be revisited if a specific, concrete need arises that cannot be met otherwise, not as a "nice to have" architectural generalization. Treat any future proposal to add this as requiring an explicit, separate justification, not an assumed roadmap item.

- **Formal interface contracts between modules.** MLflow's "flavors" pattern (typed manifest + signature + storage URI + generic fallback) was identified as the likely model to adopt, but building this now would be speculative — there currently isn't a concrete pair of modules that need to hand data to each other in a way that justifies a formal contract. Build this once at least two real modules have an actual, specific interoperability need, not preemptively.

- **vLLM / TensorRT-LLM / CUDA-datacenter LLM serving and quantization for LazyTune.** The original LazyTune research was cloud-serving-oriented; v1 rescoped this down to local-serving formats (GGUF, MLX) appropriate for the MPS-primary, local-only boundary. The datacenter-serving tooling remains a valid future direction if the platform ever adds a serving/deployment tier, but is out of place in a local-first v1.

- **Populating the actual Tier 1/Tier 2 model checkpoint allowlist per module.** This is not a one-time design task — the allowlist structure is locked, but the actual list of specific model checkpoints in each tier, per module, is an ongoing maintenance responsibility that needs continuous upkeep as new models are released and existing ones' license terms change (see also Part 5, item 3, for the legal re-verification angle on this same task).

- **Linux+CUDA validation as the secondary backend target.** MPS is primary and is where v1 implementation effort is concentrated. Linux+CUDA is designed for, in the sense that the architecture shouldn't structurally preclude it, but it has not been validated end-to-end. This validation pass should happen once the MPS-primary implementation is stable, not in parallel with it.

This appendix exists to guarantee nothing substantive from the nine original research documents was silently dropped — it lists specific mechanisms, failure-mode analyses, and proposed capabilities that exist in the source material but weren't spelled out in the main body. None of these change any decision already locked in Part 4; they are additional design detail to fold in when each module is actually implemented, and the original per-module .docx research remains the authoritative source for full derivations, formulas, and code.

**AutoML** — The original research's motivating diagnosis for why AutoML tools get abandoned in production is not restated in Part 3: AutoML tooling generally automates only the "last 20%" of the ML pipeline (model selection/tuning) while the "first 80%" (feature engineering, table joins, temporal modeling, data cleaning) remains manual; multi-layer stacked ensembles (as used by AutoGluon/PyCaret) introduce unacceptable real-time latency for production serving; and pickle-based serialization is fragile across environment/version drift between training and serving. This diagnosis is the reasoning behind the module's three signature features (streaming/incremental fit, drift detection, ONNX compile), which are documented in Part 3 — the diagnosis itself is preserved here for completeness.

**LazyClean** — The original research proposes a third failure-mode mitigation not carried into Part 3: **streaming/incremental label-error detection under threshold drift**, combining Dynamic Quantile Buffering (a sliding-window FIFO buffer of out-of-sample class-conditional probabilities, with class thresholds updated continuously via running quantile estimators as the data distribution shifts) and Conformal Margin Tracking (flagging incoming samples as label errors when their confidence margin falls below a dynamically-calibrated conformal quantile threshold). This enables real-time label auditing on streaming data without full-dataset retraining, and should be treated as an additional LazyClean capability, not just the batch-mode DeCoLe/D4 pipeline described in Part 3.

**LazyGraph** — The original research's "Dynamic and Streaming Graph Vulnerabilities" analysis is not carried into Part 3. It identifies three distinct production failure modes relevant if/when LazyGraph moves beyond static batch benchmarking: (1) continuous sparse-adjacency reconstruction — CSR/COO index arrays must be rebuilt on the CPU and re-uploaded to the GPU at every sliding-window step in a dynamic graph, creating a PCI-Express transfer bottleneck; (2) CUDA JIT "compilation storms" — dynamic node/edge-shape changes force constant kernel recompilation (e.g., via `torch.compile`), producing multi-second latency spikes that break real-time streaming pipelines; (3) GPU memory fragmentation — repeatedly allocating/deallocating differently-sized sparse arrays can trigger OOM crashes even when total free VRAM is theoretically sufficient. These are lower near-term priority given v1's local/batch benchmarking scope, but should be tracked for any future streaming/production LazyGraph deployment.

**LazyTune** — Two mitigations from the original research are not carried into Part 3: (1) **Frobenius-norm-based early stopping** — monitoring the classifier-head gradient's Frobenius norm over a short tail window during full training runs (distinct from and complementary to the BOHB/Multi-Power-Law micro-tuning system already described), since higher gradient norms correlate with loss instability and poor generalization, allowing early pruning of bad hyperparameter configurations; and (2) **multi-GPU dependency-breakage risks** — ABI mismatches between compiled CUDA/Triton extensions and PyTorch/driver versions (causing segfaults or silent memory corruption), FlashAttention-3 backward-pass gradient bugs that can break RL/GRPO convergence, and Hybrid KV-Sharing architecture breakage (e.g., Gemma-4-class models) when `use_cache=False` disrupts inter-layer state during fine-tuning. Both are lower near-term priority given v1's MPS-primary, local-model, single-device scope, but apply directly once the deferred CUDA/multi-GPU path (Part 6) is revisited.

**LazyRed** — Three findings from the original research are not carried into Part 3: (1) **Long-horizon multi-turn state fragmentation** — context-window saturation from "transcript concatenation" defenses (quadratic-cost re-evaluation of the full chat history at every turn, broken by sliding-window truncation), the "conversational safety gap" (a target's partial safety refusal mid-dialogue resets attacker state and stalls automated multi-turn attacks that lack backtracking/recovery), and decoupled agent-state desynchronization (attacker and target loops drifting out of sync with an autonomous agent's internal planning/memory state); (2) **token-cost overheads in automated optimization sweeps** — Tree-of-Attacks-style search has exponential token/API cost growth with branching factor and search depth, mitigated via evaluator-driven pruning (cutting up to ~60% of target queries before they're sent), Adversarial Prompt Distillation (offloading mutation generation to a fine-tuned small local model instead of an expensive large one), and Prompt Embedding Optimization (white-box continuous-embedding optimization that avoids token expansion entirely); (3) **measurement instability mitigations** — false positives (LLM-judge position/verbosity bias, safety-refusal-language false-flagging) and false negatives (semantic obfuscation via ROT13/Base64/multilingual encoding bypassing keyword-based detectors) are addressed via Multi-Model Jury Consensus (a heterogeneous-model panel voting on each verdict), Structured Binary Rubrics (converting open-ended grading into consistent binary classification), and continuous calibration against human-verified golden datasets. All three are directly relevant to the Guardrail/Firewall Layer and evaluation engine described in Part 2 (§2.4) and Part 3, and should inform their detailed design.

**LazyAgent** — The original research's RAG-pipeline-tuning bottleneck analysis is only partially carried into Part 3 (stage-cascade failure is mentioned; the rest is not). In full: RAG pipeline tuning faces a combinatorial search-space explosion (parameters across chunking, indexing, reranking, and model choice scale exponentially), reranker latency tradeoffs (cross-encoder rerankers such as MonoT5 or BGE FlagReranker meaningfully improve accuracy but add real inference latency that must be budgeted), and production vector-database disconnects (parameters tuned against local dev engines like Chroma/LanceDB can degrade once deployed against production engines like Qdrant/Milvus, due to network latency, sharding topology, and index lock contention under concurrent writes). These are the specific failure modes the module's DISCO-based sample-condensation and multi-objective Pareto optimization (documented in Part 3) are designed to address.

**LazyEdge** (deferred module) — The original research's detailed embedded-execution bottleneck mechanics are condensed to a single sentence in Part 3. In full, for reference whenever this module is un-deferred: silent CPU-fallback subgraph fragmentation (memory-copy latency, context-switching overhead, and memory-layout realignment penalties when unsupported operators force a graph split between NPU and CPU execution); simulated-vs-physical quantization divergence (accumulator overflow in narrow fixed-point hardware registers, formally bounded by the Colbert et al. minimum-accumulator-width formula; dyadic fixed-point scale-conversion rounding-error divergence not modeled during training; SIMD register-packing/alignment penalties on Cortex-M/CMSIS-NN targets); and flash/SRAM memory-mapping conflicts (heap fragmentation from dynamic allocation on bare-metal microcontrollers with no virtual memory; Execute-in-Place flash-resident weight storage to preserve volatile SRAM for activations only; mmap-based weight paging on MMU-enabled edge devices). These mechanics are fully preserved in the original LazyEdge source document and should be pulled from there directly rather than re-derived from Part 3's summary when this module is prioritized.
